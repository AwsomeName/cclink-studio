# 手动 Git 备份

> 状态：规划中，尚未实现
>
> 最后更新：2026-07-17

## 结论

CCLink Studio 第一版只提供“手动备份到 Git”，不做自动备份，也不做完整 Git 客户端。

用户只需要在设置中保存一次 GitHub 账号和访问 Token，然后在当前本地工作空间点击“备份到 Git”。项目第一次备份时，用户可以填写完整远程仓库地址，也可以只填写项目名；完整地址直接使用，项目名则在用户 GitHub 账号下创建同名私有仓库。后续备份复用已经绑定的远程仓库。

这是用户自有 Git 仓库上的本地项目备份能力，不是 CCLink 官方账号、云同步、网络工作空间或官方发布链路。未配置 Git 备份时，Studio 的本地工作台、Agent、浏览器、编辑器、Terminal 和设备能力必须照常可用。

## 产品目标

第一版只解决三个问题：

- 用户可以配置自己的 GitHub 账号和访问 Token。
- 本地工作空间可以绑定一个用户指定或自动创建的远程仓库。
- 用户点击一次“备份到 Git”，Studio 自动提交当前项目变更并 Push。

成功标准不是让用户在 Studio 中管理 Git，而是让不熟悉 Git 命令的用户也能把当前项目手动备份到自己的远程仓库。

## 明确不做

第一版不做：

- 文件监听、定时备份、退出时备份或任何后台自动 Push。
- 多用户、多 GitHub 账号或项目级账号切换。
- Git 分支管理、暂存区管理、提交历史、Diff 编辑器或 Remote 管理器。
- Pull、Fetch、Merge、Rebase、冲突解决或 Force Push。
- 自动覆盖非空远程仓库的历史。
- GitHub 之外平台的“按项目名自动创建仓库”。
- 项目恢复、双向同步、云文件树或网络工作空间。
- CCLink 官方账号与 GitHub 账号绑定。

完整远程地址模式可以使用本机 Git 已支持的 HTTPS 或 SSH 地址；如果使用非 GitHub 服务，Studio 不负责自动创建仓库，认证能力以用户本机 Git/SSH 环境为准。

## 用户交互

### 设置 Git 备份

设置页新增“Git 备份”区域：

```text
Git 备份

Git 状态：已安装 Git 2.x
GitHub 账号：[ zhangsan       ]
访问 Token：[ •••••••••••••• ]

[测试连接] [保存]
```

交互规则：

- GitHub 账号是普通非敏感设置，可以回显。
- Token 只允许覆盖或清除，不允许 Renderer 读回明文。
- “测试连接”只验证账号和 Token 当前有效，不创建仓库；建仓和写入权限在实际操作时验证。
- 本机没有 Git 时显示安装提示，但不能阻断 Studio 启动。
- 本机加密存储不可用时拒绝保存 Token，不允许降级为明文。

第一版不要求用户填写提交姓名和邮箱。Studio 生成的备份提交使用固定身份：

```text
CCLink Studio Backup <backup@cclink.local>
```

该身份只通过本次备份进程的 Git 配置或环境变量传入，不写入仓库配置，也不修改用户的全局 `~/.gitconfig`。

### 第一次备份

当前本地工作空间提供一个“备份到 Git”按钮。第一次点击时打开最小对话框：

```text
备份到 Git

远程仓库地址或项目名：
[ my-project                         ]

[取消] [备份]
```

输入解析规则：

- `https://github.com/owner/repo.git`、`ssh://...` 或 `git@host:owner/repo.git` 识别为完整远程地址。
- `my-project` 识别为 GitHub 项目名。
- 项目名只允许 GitHub 仓库名可接受的安全字符，禁止路径、Shell 片段和控制字符。
- 输入项目名时，通过 GitHub API 在设置账号下创建同名私有仓库；同名仓库已经存在且用户有写权限时直接绑定。
- 自动创建的仓库必须是私有仓库，第一版不提供公开选项。
- 输入完整地址时不创建仓库，也不改变远程仓库可见性。

备份成功后，Studio 按本地 `projectId` 记录远程仓库绑定。项目移动后继续沿用原绑定；项目复制并生成新 `projectId` 后不得继承原项目绑定。

### 后续备份

项目已经绑定远程仓库后，用户再次点击“备份到 Git”不再询问地址：

```text
[备份到 Git]

上次备份：2026-07-17 20:30
远程仓库：github.com/zhangsan/my-project
```

操作结果只需要四类状态：

- `备份成功 · 刚刚`
- `没有需要备份的变更`
- `正在备份…`
- `备份失败：<可操作原因>`

远程绑定需要提供“更换远程仓库”入口，但它属于按钮的次级菜单，不扩展成 Remote 管理器。

## 一次备份的行为

用户点击“备份到 Git”后，主进程按以下顺序执行：

1. 确认当前对象是合法、存在、可读写的本地工作空间。
2. 检测本机 Git；缺失时立即返回可操作错误。
3. 读取当前项目的远程绑定；首次备份则解析用户输入并创建或绑定仓库。
4. 项目不是 Git 仓库时执行 `git init`，默认分支使用 `main`。
5. 为本次备份命令注入固定提交身份，不修改仓库或全局 Git 配置。
6. 写入 Studio 本地排除规则，并在提交前执行敏感文件检查。
7. 将当前项目变更加入暂存区。
8. 没有文件变化但存在尚未 Push 的本地提交时继续 Push；两者都没有时返回“没有需要备份的变更”。
9. 有文件变化时创建备份提交，格式为 `CCLink backup YYYY-MM-DD HH:mm:ss`。
10. 配置或校验备份远程地址，然后 Push 当前分支。
11. 成功后记录时间和远程地址；失败时保留本地提交并返回可诊断错误。

第一版的“备份”明确表示提交当前项目的全部可备份变更。它不是用户手工 Git 工作流的精细提交工具。如果项目已有暂存内容，备份提交会包含这些内容；UI 文案必须明确“备份当前全部变更”。

## 文件排除与敏感信息

自动创建 Git 仓库时，Studio 使用 `.git/info/exclude` 写入本机排除项，不擅自修改用户项目的 `.gitignore`。

默认排除至少包括：

- `.cclink-studio/`
- `node_modules/`
- 常见构建输出和缓存目录。
- `.env`、`.env.*`。
- 常见私钥、证书私钥和 SSH 密钥文件。

仅依赖 ignore 规则不够：已经被 Git 跟踪的敏感文件不会因新增 exclude 而消失。因此每次提交前必须检查将要提交的路径；发现 `.env`、私钥或明显凭证文件时阻止 Push，并提示用户自行移除或确认项目现有 Git 规则。第一版不提供“忽略警告继续上传”。

大文件、Git LFS、子模块和嵌套仓库不做自动修复。Git 或远程平台拒绝 Push 时，返回错误和诊断信息，不改写项目结构。

## 认证和安全边界

- Token 使用 Electron `safeStorage` 加密后写入独立凭证文件，不进入 `settings.json`、工作空间文件或日志。
- Renderer 只能看到 `configured: boolean` 等脱敏状态，不能读取 Token 明文。
- 匹配已配置 GitHub 账号的 HTTPS Push 通过受控的 `GIT_ASKPASS` 机制向 Git 子进程提供凭证，Token 不拼接进远程 URL、命令参数或错误文本；其他 HTTPS 地址使用本机 Git 凭证机制。
- SSH 地址继续使用用户本机 SSH Agent 和密钥，不读取或托管 SSH 私钥。
- Git 子进程使用参数数组直接启动，禁止通过 Shell 拼接账号、项目名、路径或远程 URL。
- 主进程只允许对当前已解析的本地工作空间执行操作，并沿用工作空间路径边界检查。
- GitHub API 只访问用户明确配置的平台，不依赖 CCLink 官方 API 或官方账号。
- 日志记录阶段、退出码和脱敏错误，不记录 Authorization Header、Token 或带凭证 URL。

非快速前进、远程已有不同历史、权限不足或认证过期时，第一版只停止并报错，绝不自动 Pull、Merge、Force Push 或删除远程内容。

## 数据与服务边界

建议的数据分层：

```text
settings.json
└── gitBackupProvider / gitBackupUsername

userData/git-backup/secrets.enc
└── safeStorage 加密后的访问 Token

userData/git-backup/projects.json
└── projectId → remoteUrl / lastBackupAt
```

- 非敏感账号信息进入普通设置。
- Token 只进入加密凭证存储。
- 项目绑定按稳定 `projectId` 保存，不把 Token 或远程凭证写进项目目录。
- Git 自身的 `.git/config` 可以保存无凭证远程地址，但不能保存内嵌 Token 的 URL。

主进程新增 `GitBackupService`，负责 Git 检测、GitHub 连接测试、仓库创建、项目绑定、敏感文件预检、Commit 和 Push。Renderer 只通过白名单 preload API 调用，不直接访问 Node.js、Git CLI 或凭证。

## 开发计划

### M1：共享契约和安全凭证

#### 目标

建立 Git 备份的数据模型、IPC 边界和 Token 安全存储，让后续功能不把凭证混入普通设置或 Renderer。

#### 方案

- 在 `src/shared/ipc/` 定义 Git 状态、账号状态、项目绑定、备份输入、进度和结果类型。
- 在普通设置中增加 GitHub provider 和 username；Token 不进入 `AppSettings`。
- 抽取或复用 `safeStorage` 加密模式，新增独立 `GitBackupCredentialStore`。
- preload 只暴露账号状态、保存/清除 Token、连接测试和备份操作，不暴露明文凭证。
- 给所有 IPC 输入增加运行时校验和长度限制。

#### 验收标准

- [ ] Token 不出现在 `settings.json`、项目目录、Renderer 状态、日志和 IPC 返回值中。
- [ ] 本机加密不可用时保存 Token 明确失败，不生成明文兜底文件。
- [ ] 重启应用后可以得到“已配置”状态并成功使用凭证，但 Renderer 仍无法读回 Token。
- [ ] 非法 provider、账号、项目名和超长输入被主进程拒绝。
- [ ] 未配置 Git 备份时 Studio 可以正常独立启动。

### M2：Git 执行内核

#### 目标

提供受控、可测试的本机 Git 执行能力，覆盖检测、初始化、状态、提交和 Push 所需的最小命令集。

#### 方案

- 新增 Git 可执行文件检测和版本解析。
- 新增只使用参数数组的进程执行器，支持工作目录、超时、退出码和脱敏输出。
- 禁用交互式终端提示，通过受控环境变量处理 HTTPS 认证。
- 实现仓库检测、`git init`、进程级提交身份、远程读取/设置、变更检测和待 Push 判断。
- 沿用本地工作空间 realpath、主目录边界和目录存在性检查。

#### 验收标准

- [ ] Git 缺失、不可执行或版本读取失败时返回稳定错误码，不阻断应用启动。
- [ ] 路径、账号和远程 URL 中的空格或特殊字符不会转化为 Shell 命令执行。
- [ ] Git 命令超时或退出异常时子进程被清理，按钮可以再次使用。
- [ ] 初始化和备份提交不会修改仓库身份配置或全局 `~/.gitconfig`。
- [ ] Git 输出中的 Token 和带凭证 URL 会被脱敏。
- [ ] 单元测试覆盖成功、无变更、认证失败、非 Git 目录和非法路径。

### M3：设置页 Git 备份配置

#### 目标

让单个本机用户完成一次账号配置，并清楚知道 Git、账号和 Token 是否可用。

#### 方案

- 设置页新增“Git 备份”区域和搜索索引。
- 展示 Git 检测状态、GitHub 账号、遮罩 Token 输入、“测试连接”“保存”“清除凭证”。
- 测试连接调用 GitHub 用户接口验证凭证当前有效，只返回脱敏账号摘要和错误；具体建仓和写入权限在实际操作时验证。
- 保存成功后清空前端 Token 输入，只保留“已配置”状态。

#### 验收标准

- [ ] 用户可以保存账号和 Token，重启后仍显示已配置。
- [ ] 测试连接成功时显示匹配的 GitHub 账号。
- [ ] Token 错误、过期、权限不足和网络失败有不同的可操作提示。
- [ ] 修改账号不会意外复用一个明显不匹配的 Token，必须重新测试或重新保存。
- [ ] 清除凭证后无法创建仓库或执行需要该 Token 的 HTTPS Push。

### M4：项目绑定和首次备份入口

#### 目标

让当前本地工作空间用一个输入框绑定完整远程地址，或按项目名创建 GitHub 私有仓库。

#### 方案

- 在当前工作空间增加单一“备份到 Git”入口和简洁状态展示。
- 首次点击时展示“远程仓库地址或项目名”输入框。
- 实现完整 HTTPS/SSH 地址与安全项目名的确定性解析。
- 项目名模式调用 GitHub API：同名仓库存在且可写则绑定，不存在则创建私有仓库。
- 绑定按本地 `projectId` 持久化，提供次级“更换远程仓库”操作。

#### 验收标准

- [ ] 完整 HTTPS 地址、SSH 地址和纯项目名能被正确区分。
- [ ] 项目名模式只创建私有仓库。
- [ ] 同名可写仓库可以直接绑定；无权限或同名冲突不会创建带随机名称的意外仓库。
- [ ] 绑定成功后再次点击不再询问地址。
- [ ] 项目移动后保留绑定，项目复制并获得新 `projectId` 后不继承绑定。
- [ ] 取消对话框不会初始化 Git、创建仓库或写入绑定。

### M5：手动备份闭环

#### 目标

一次按钮点击完成敏感文件预检、提交当前全部可备份变更并 Push，且失败时不破坏远程历史。

#### 方案

- 首次备份时初始化 Git、为备份命令注入固定身份并写入 `.git/info/exclude`。
- 提交前枚举候选路径并阻止明显敏感文件进入提交。
- 执行 `git add -A`，有变化时生成带时间戳的备份提交。
- 没有工作区变化但存在未 Push 提交时继续 Push。
- Push 只允许正常快进；拒绝自动 Pull、Merge、Force Push 和远程删除。
- 成功后保存 `lastBackupAt`，失败时保留本地提交和脱敏诊断。

#### 验收标准

- [ ] 新目录可以初始化、提交并 Push 到自动创建的私有仓库。
- [ ] 已有 Git 项目可以备份到用户填写的可写远程地址。
- [ ] 第二次无变化时不产生空提交，并显示“没有需要备份的变更”。
- [ ] 本地已有未 Push 提交时，即使工作区干净也会尝试 Push。
- [ ] `.env`、私钥和明显凭证文件在提交前被阻止。
- [ ] 认证失败、离线、大文件拒绝、远程非快进和远程历史冲突不会触发强推或覆盖。
- [ ] Push 失败后本地文件和本地提交仍然存在，可以再次手动备份。

### M6：状态、诊断和完整验收

#### 目标

让用户能够理解一次手动备份是否完成，并完成跨进程、重启和 OSS 边界验收。

#### 方案

- UI 只展示空闲、备份中、成功、无变化和失败状态。
- 备份中禁用重复点击；操作结束后恢复按钮。
- 失败提示提供可复制的脱敏诊断，不显示 Token。
- 增加主进程服务、凭证存储、IPC、设置 Store 和 UI 的测试。
- 更新功能文档、开发文档和本地 smoke 清单。

#### 验收标准

- [ ] 用户只通过“配置一次账号 → 首次填地址或项目名 → 点击备份”完成闭环。
- [ ] 应用重启后账号配置状态、项目绑定和上次备份时间正确恢复。
- [ ] 重复点击不会并发创建多个提交或 Push 进程。
- [ ] 错误诊断可复制且不包含 Token、Authorization Header 或带凭证 URL。
- [ ] 未配置账号时，SSH 完整地址仍可使用本机 SSH 认证；项目名模式给出配置账号提示。
- [ ] `pnpm typecheck`、`pnpm lint`、`pnpm test`、`pnpm build` 和 `pnpm verify:oss-boundary` 通过。
- [ ] 无 Git、无网络、无 GitHub 配置时 Studio 仍能免登录启动并使用其他本地能力。

## 里程碑顺序和交付判断

里程碑必须按 M1 → M6 推进。M1 和 M2 是安全与执行底座；在它们完成前，不应先做一个会把 Token 放进普通设置、或通过 Terminal 字符串拼接运行 Git 的 UI 原型。

每个里程碑只有在对应自动化测试和人工验收同时通过后才算完成。M4 创建出远程仓库但 M5 不能可靠 Push，不算用户价值闭环；M5 能 Push 但凭证可能进入日志，同样不能交付。

## 人工验收场景

最终至少完成以下真实场景：

1. 全新目录，输入项目名，自动创建私有仓库并完成首次备份。
2. 已有 Git 项目，输入完整 HTTPS 地址并完成备份。
3. 已有 Git 项目，输入 SSH 地址并使用本机 SSH Agent 完成备份。
4. 无文件变化时再次点击，不产生空提交。
5. Token 失效、网络中断和远程非快进时只报错，不覆盖远程。
6. 项目包含 `.env` 或私钥候选文件时阻止提交。
7. 重启 Studio 后继续使用已绑定仓库完成下一次手动备份。
8. 未安装 Git 时 Studio 正常启动，备份入口给出降级说明。

## /grilling

结论先说：第一版可以保持非常简单，但“简单”只能体现在用户交互，不能靠明文 Token、Shell 字符串拼接或强推远程来换取。

最关键的假设是“备份当前全部变更”符合用户预期。它会把已有暂存内容和未跟踪文件一起提交，因此按钮和首次说明必须用“备份当前全部变更”，不能伪装成精细 Git 提交。

最危险的失败路径不是 Push 失败，而是敏感文件已经进入提交或日志。敏感路径预检、加密凭证和日志脱敏属于第一版交付门槛，不是后续优化。

项目名模式并不是通用 Git 能力，它依赖 GitHub API。第一版要么明确只支持 GitHub 自动创建，要么删除项目名模式；不能用“Git 账号”模糊不同托管平台的授权和建仓差异。

远程已经存在不同历史时，自动合并或强推都会把简单备份升级成同步系统。第一版应停在错误提示，让用户换用空仓库或自行处理远程历史。
