# DeepInk 创始人项目运营工作台需求记录

> 状态：2026-07-14 需求记录与优先级重排；项目内运营助手 M7.1-M7.5 已实现待验收
> 背景：DeepInk 先服务开发者本人正在推进的多个真实项目，再从这些高频工作流中提炼通用产品能力。
> 关联文档：`docs/features/product-milestones.md`、`docs/features/project-operations-assistant.md`、`docs/features/ai-work-browser.md`、`docs/features/project-system.md`、`docs/features/terminal-tab-model.md`

## 结论

DeepInk 下一阶段的第一用户不是抽象的“知识工作者”，而是创始人自己：

```text
用 DeepInk 维护多个真实项目的上线、内测、宣发、远程调试和日常运营。
```

这会改变近期优先级：

1. **远程项目支持升为 P0**：远程工作空间、远程文件、远程 Terminal / Agent 执行必须尽快形成闭环。
2. **项目内运营助手升为 P0 工作流**：在项目目录里维护平台配置、写 Markdown 文案、用浏览器提交平台内容，是 DeepInk 最应该先打穿的可用场景。
3. **Android 战略收缩**：不再推进本地虚拟机，也不再考虑云手机；后续只保留用户自有手机的 USB / Wi-Fi 连接方向。
4. **专业项目支持先走通用工作空间**：AI 眼镜、电路板、PCB、嘉立创下单、游戏资产管理等，不先做垂直大功能，先用浏览器、文件、Markdown、远程执行和资料管理承接。

## 需求 1：Android 功能抽离与封存

新的产品决策：

- 不再默认调用 Android 虚拟机。
- 不再继续推进 Google AVD / QEMU 模拟器生命周期管理。
- 不再考虑云手机作为 DeepInk 的移动自动化方向。
- 现有 Android 相关代码需要抽离、封存、降级为可选能力，避免在主流程中被误触发。
- 后续只考虑用户自己的真实手机：
  - USB 连接。
  - Wi-Fi ADB 连接。
  - 必要时保留 scrcpy 投屏 / ADB 操控思路，但目标设备是真机，不是模拟器或云实例。

产品边界：

- Android 不再是近期“第二自动化支柱”。
- Android 不再作为 v0.1 可用状态的验收项。
- 设置页可以保留“设备”诊断入口，但不应引导用户安装 SDK、创建 AVD 或开通云手机。
- Agent 不应在没有用户明确选择设备的情况下调用 Android 工具。

第一阶段实现建议：

- 把模拟器生命周期相关代码标记为 legacy / archived。
- UI 中隐藏或弱化 AVD 安装、创建、启动入口。
- MCP Android 工具默认不可用，除非检测到用户主动连接的真机。
- capability 状态明确显示：`Android emulator archived; physical device not connected`。

## 需求 2：项目内运营助手

近期最真实的高频工作流：

```text
打开项目 → 读取项目资料 → 写 Markdown 文案 → 打开平台页面 → 填写提交 → 写回发布记录
```

修正后的产品判断：

- 不新增独立“运营平台”。
- 不把每个平台做成工作区。
- 不第一版做密码库或完整账号管理器。
- 工作区仍然是项目目录。
- 平台信息只是项目里的 `deepink-accounts.json`。
- 文案由工作会话写入项目 Markdown。
- 平台操作由另一个工作会话打开浏览器、读取平台配置、填表并等待确认。
- 发布记录先写回项目 Markdown，例如 `docs/发布记录.md`。

推荐项目结构：

```text
项目目录
├─ README.md
├─ docs/
│  ├─ 宣发方案.md
│  ├─ 公众号首发稿.md
│  ├─ 知乎版本.md
│  └─ 发布记录.md
└─ deepink-accounts.json
```

`deepink-accounts.json` 只保存轻量信息：

```json
{
  "version": 1,
  "platforms": [
    {
      "id": "wechat-mp",
      "name": "微信公众号",
      "url": "https://mp.weixin.qq.com",
      "account": "DeepInk",
      "notes": "扫码登录；发布前必须人工确认。",
      "browserProfile": "wechat-mp"
    }
  ]
}
```

密码、Token、恢复码等秘密信息不进入项目可见文件；后续如支持密码，只保存加密凭据引用。

详见 `docs/features/project-operations-assistant.md`。

## 需求 3：支持其他真实项目

除 CCLink 外，还有多类项目需要 DeepInk 承接。

### AI 眼镜 / 硬件项目

涉及：

- 电路板资料。
- 原理图。
- PCB。
- BOM。
- 嘉立创下单。
- 供应链页面和订单状态。
- 技术资料整理。

DeepInk 第一阶段不需要做 EDA 工具，也不需要直接解析复杂 PCB 工程。应该先支持：

- 资料和文件归入工作空间。
- 浏览器打开嘉立创、供应商、数据手册网站。
- Markdown 记录需求、BOM、下单步骤和问题。
- Agent 辅助核对网页表单、整理资料、生成检查清单。

### 游戏项目

涉及：

- 游戏资产维护。
- 素材目录管理。
- 发布资料。
- 远程构建或服务器维护。

第一阶段应通过工作空间文件、浏览器、Markdown、远程 Terminal 和远程会话承接，不先做专门资产管理系统。

### 远程服务器上的项目

这是第三类需求里最重要、最着急的一项。

DeepInk 需要支持：

- 添加远程工作空间。
- 浏览远程文件树。
- 打开远程文件。
- 运行远程命令。
- 查看远程日志。
- 让 Agent / Codex / 自定义后端在远程上下文里工作。
- 对远程命令、文件写入、部署操作做权限确认和审计。
- 远程错误能定位到账号、transport、远端 Agent、文件 provider、执行后端或权限层。

结论：**远程项目支持应该成为下一阶段 P0 主线**，优先级高于 Android、IM、云盘、记忆系统和垂直行业集成。

## 近期 MVP 工作流

### MVP-A：远程项目维护

目标：用 DeepInk 打开一个远程服务器项目，并完成真实维护动作。

验收：

- 能添加一个远程工作空间。
- 能看到远程文件树或明确失败原因。
- 能打开远程文件为只读或可编辑 Tab。
- 能打开远程 Terminal。
- 命令执行有权限确认和审计。
- Agent 能围绕当前远程工作空间回答问题或执行受控操作。
- 断线、权限失败、远端 Agent 不兼容时能展示结构化错误。

### MVP-B：项目内运营助手

目标：用 DeepInk 完成一次真实宣发动作。

验收：

- 项目里有 `deepink-accounts.json`。
- 文案会话能读取项目资料并写 Markdown。
- 平台操作会话能读取平台配置和目标 Markdown。
- 浏览器能使用平台对应 profile 打开目标网站。
- Agent 帮助复制、填写、上传或准备提交内容。
- 真正发布、提交、删除、改资料前必须确认。
- 发布后把 URL、截图、状态和操作记录保存到 `docs/发布记录.md`。
- 后续能打开评论或反馈页面，读取并追加总结到项目文档。

当前状态：

- 已实现项目账号配置模板、读取和校验。
- 已实现文案草稿创建和文案工作会话入口。
- 已实现平台浏览器 profile 打开入口。
- 已实现平台操作工作会话入口。
- 已实现发布记录追加到项目 Markdown。
- 待真实平台人工验收。

### MVP-C：平台浏览器 Profile

目标：让项目配置中的平台入口能复用登录态。

验收：

- `deepink-accounts.json` 能声明 `browserProfile`。
- 不同平台可以使用不同 profile。
- 重启后浏览器 profile 仍保留 Cookie 和站点状态。
- UI 能提示当前页面正在使用哪个 profile。

### MVP-D：真实手机连接

目标：只支持用户自己的手机，不支持模拟器和云手机。

验收：

- 能检测 USB 连接设备。
- 能连接 Wi-Fi ADB 设备。
- 能显示设备状态。
- Agent 只有在用户明确选择设备后才能调用 Android 工具。
- 未连接设备时 Android capability 明确降级，不影响浏览器、Markdown 和远程能力。

## 明确暂缓

- 暂缓本地 Android 模拟器。
- 暂缓云手机。
- 暂缓完整 IM 产品。
- 暂缓自建云盘产品。
- 暂缓完整 AI 记忆系统。
- 暂缓 EDA / PCB / 游戏资产的垂直编辑器。
- 暂缓把 DeepInk 做成通用社媒管理 SaaS。
- 暂缓密码库和全自动登录。

## /grilling

结论先说：这些需求很合理，而且比“做一个一站式 AI 桌面服务”更能逼出可用产品。但风险也更直接。

第一，远程项目支持不是“再加一个远程入口”。它要求工作空间、文件 provider、Terminal、Agent runtime、权限、审计、错误模型同时闭环。只要其中一个环节空着，用户就会觉得远程能力是假的。

第二，项目内运营助手不是“运营平台”。真正难点是把项目资料、Markdown 草稿、浏览器登录态、提交前确认和发布记录串起来。Agent 自动发错内容，比功能不可用更糟。

第三，Android 必须真的收缩。只要模拟器和云手机入口还留在主流程里，产品优先级就会继续被稀释。封存不是删代码，而是让它不再定义近期路线。

第四，AI 眼镜、PCB、游戏资产这些项目不要急着垂直化。先让 DeepInk 成为“项目现场”：资料、网页、文档、远程机器、会话都能围绕同一个工作空间组织起来。垂直能力以后从真实重复操作里长出来。

下一步最该做的不是继续扩大功能面，而是选择两个真实任务做验收：

1. 用 DeepInk 完成一次 CCLink 宣发发布。
2. 用 DeepInk 连接并维护一个远程服务器项目。

如果这两件事做不顺，DeepInk 还不能叫可用；如果这两件事稳定了，其他项目会自然被吸进工作空间模型里。
