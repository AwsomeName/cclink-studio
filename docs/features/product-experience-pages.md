# DeepInk 产品体验页面定义

> 状态：产品形态共识稿
> 目的：先定义核心页面的职责边界，避免在 UI 上继续堆功能；细节页面和具体功能后续再拆。
> 迁移盘点：见 `docs/features/ui-entry-migration-audit.md`  
> 里程碑验收：见 `docs/features/product-milestones.md`
> 远程错误模型：见 `docs/features/remote-error-model.md`
> Terminal 模型：见 `docs/features/terminal-tab-model.md`

## 结论

DeepInk 的第一层产品形态应当收敛为：

```text
Activity Bar
  ↓ 选择一种内容库或系统区
Sidebar
  ↓ 浏览当前内容库
Workbench Tabs
  ↓ 承载当前操作
Right Agent Panel
  ↓ 对当前工作进行协作、执行和确认
Settings
  ↓ 管理账号、能力、设备和诊断
```

核心原则：

- **工作空间是主入口**：本地文件夹、远程目录、会话、草稿、Tab 状态都围绕工作空间组织。
- **文件树必须保留**：工作空间不是替代文件浏览，而是先选归属，再看该归属下的文件和会话。
- **能力不要都放 Activity Bar**：Android、CCLink、同步这类能力重要，但不是日常一级内容库。
- **账号和连接问题进设置**：导入旧账号、同步服务器、设备安装、诊断日志不应占据日常侧栏。
- **Remote 不等于 CCLink**：Remote 是 DeepInk 的远程工作能力，CCLink 只是连接路线之一。
- **Tab 是工作现场**：浏览器、Markdown、Android、Terminal、远程文件、设置、预览都是 Tab 类型。

## 产品模型：Remote / Transport / Codex / Terminal

这次产品逻辑的关键变化是：把 Remote 从 CCLink 里解耦出来。

```text
DeepInk
├─ 工作空间
│  ├─ 本地工作空间
│  ├─ 远程工作空间：直连
│  └─ 远程工作空间：CCLink
│
├─ 连接通道 Transport
│  ├─ local
│  ├─ direct：IP / 域名 / 内网 / VPN / SSH / 远端 Agent 地址
│  └─ cclink：账号 / 配对 / 实时链路 / 中继 / 设备发现
│
├─ 执行后端
│  ├─ DeepInk Agent
│  ├─ Codex
│  ├─ Claude Code
│  └─ 自定义后端
│
└─ 工作 Tab
   ├─ Markdown
   ├─ Browser
   ├─ Android
   ├─ Terminal
   └─ Remote File / Session / Preview
```

产品规则：

- Remote 是“工作空间可以在远端”的能力，不是 CCLink 的别名。
- CCLink 是 Remote Transport 的一种，主要解决账号配对、设备发现、NAT/公网不可达和移动端协同。
- 直连 Remote 也必须成立：用户拥有 IP、域名、内网地址、VPN、SSH 或远端 Agent 地址时，可以不经过 CCLink 打开远程工作空间。
- Codex 是执行后端，不是工作空间类型；本地和远程工作空间都可以选择 Codex、DeepInk Agent、Claude Code 或自定义后端。
- Terminal 是工作空间里的 Tab 类型；本地工作空间打开本机 shell，远程工作空间打开远端 shell。
- 本地和远程仍然只作为工作空间 badge，不新增“本地/远程/服务器”一级目录。
- 远程错误遵循统一 `RemoteError` 模型：错误码描述失败语义，context 描述 transport、设备、工作空间和操作现场。

## 迁移评估与工作方案

这次迁移不是换皮，而是把“能力面板堆叠”收敛为“工作空间 + Tab + 设置”的稳定产品骨架。

优先顺序：

1. **先清入口**：把 CCLink、Android 管理、同步配置和诊断从日常侧栏里移走；Activity Bar 只保留工作空间、搜索、浏览器、消息和设置。
2. **再稳工作空间**：工作空间列表平铺展示本地和远程；激活工作空间后展开文件、会话、草稿。
3. **再接远程来源**：把 CCLink 同步到的服务器/目录转成远程工作空间；预留直连 Remote，但不要为了直连先发明新侧栏。
4. **再补 Tab 能力**：Android、Browser、Markdown、Remote File、Session、Terminal 都作为 Tab 类型接入，不做独立 Activity Bar 入口。
5. **最后做诊断体验**：账号导入、配对、日志、缓存清理和链路检查进入 `设置 > 远程连接`，避免污染日常工作区。

迁移第一原则：

> 日常侧栏只承载“用户正在浏览的内容库”，不承载“能力配置、账号状态、链路诊断”。如果一个入口主要是在出问题时才打开，它就应该进设置或诊断页，而不是占据 Activity Bar。

第一阶段迁移范围：

- 移除日常侧栏里的 CCLink 配置卡片，只在设置中保留。
- 工作空间面板显示远程工作空间条目，而不是显示“远程 Agent 面板”。
- 远程工作空间下先支持文件树、只读远程文件、会话列表和服务器在线状态。
- Tab 栏 `+` 升级为新建 Tab 菜单，浏览器图标保留为高频快捷入口。
- Android 管理进入设置，Android 运行态留在 Android Tab。

暂不迁移：

- 不立刻实现完整直连 Remote 协议，只把 UI 模型和数据模型预留好。
- 不立刻实现远程 Terminal 写操作；先按 `docs/features/terminal-tab-model.md` 定义权限、审计和生命周期，再接真实 shell。
- 不把 Codex 做成单独侧栏；它只作为会话/任务执行后端出现。
- 不把所有运维诊断做成日常页面；失败时再引导进入设置诊断。

验收标准：

- 用户每天主要停留在工作空间面板，而不是 CCLink 或 Android 配置面板。
- 本地和远程工作空间看起来是同一种对象，只是 badge 不同。
- 切换工作空间时，Tab 集合、文件树和会话归属都能跟着切换。
- 没有工作空间时，用户仍能通过 Home/未归档开始草稿或会话。
- 出错时能定位到账号、连接通道、远程 Agent、文件 provider 或执行后端中的哪一层。

## 统一会话模型

DeepInk 不设计“远程 Agent 面板”这个一级产品概念。会话就是会话，本地会话和远程会话在用户心智上没有区别，只是运行环境不同。

更准确地说：DeepInk 的会话不按“是不是 AI 对话”分类，而按“是不是一个可恢复的工作现场”分类。

```text
Conversation
├─ 即时助手会话
│  ├─ 展示位置：右侧 Agent Panel
│  ├─ 用途：随手问、解释当前 Tab、临时协作
│  ├─ 生命周期：轻量、常驻、跟随当前工作
│  └─ 典型入口：右侧 Agent 面板“新会话”
│
└─ 工作会话
   ├─ 展示位置：Workbench conversation Tab
   ├─ 用途：长期任务、远程任务、可恢复工作现场
   ├─ 生命周期：归属工作空间、可关闭视图但不删除会话
   └─ 典型入口：工作空间侧栏“会话”分组
```

一句话心智：

> 右边是随手助手，中间是正式工作。

```text
[本地] DeepInk
├─ 文件
├─ 会话
└─ Terminal

[远程 · 直连] Mac mini /Users/app/project-a
├─ 文件
├─ 会话
└─ Terminal

[远程 · CCLink] supermicro /data/research
├─ 文件
├─ 会话
└─ Terminal
```

会话通过元信息区分运行环境：

- 展示形态：即时助手 / 工作会话。
- 运行位置：本机 / Mac mini / supermicro。
- 连接通道：local / direct / cclink。
- 执行后端：DeepInk Agent / Codex / Claude Code / 自定义后端。
- 状态：在线 / 离线 / 执行中 / 等待确认。

拆分规则：

- 右侧 Agent 面板默认创建即时助手会话，不占用主工作区 Tab。
- 工作空间里的会话默认是工作会话，进入对应工作空间的 `会话` 分组。
- Workbench Tab 菜单里的“工作会话”会创建本地工作会话 Tab，而不是再创建右侧即时助手会话。
- 远程工作空间会话一定打开为工作会话 Tab，因为它代表远端机器上的可恢复工作现场。
- 重要即时会话允许“打开为工作会话”，绑定当前工作空间并打开为 `conversation` Tab；第一版不从右侧列表移除，避免用户误以为会话丢失。
- 工作空间侧栏只展示绑定到当前工作空间的工作会话，不展示右侧即时助手的“最近会话”。
- 工作会话 Tab 必须显示标题、绑定工作空间、运行位置、连接通道、执行后端和状态，避免用户分不清“本地/远程/临时/长期”。
- 关闭工作会话 Tab 只关闭现场视图，不等于删除会话；删除必须是显式动作。
- Workbench 中会话使用通用 `conversation` Tab；CCLink 只是 `conversation.transport = cclink` 的实现细节。
- 远程文件进入对应远程工作空间的 `文件` 分组。
- 账号同步、旧账号导入、配对、诊断进入 `设置 > 远程连接`。
- 连接状态显示在工作空间行、Status Bar 和必要的错误提示中。
- 不保留单独的 `远程 Agent 面板`、`CCLink 面板` 或 `Transport 面板` 作为日常入口。

会话模型字段方向：

```text
conversation
├─ surface: assistant-panel | workbench-tab
├─ workspaceRef: local | remote workspace
├─ runtime
│  ├─ location: local | remote
│  ├─ transport: local | direct | cclink
│  └─ backend: deepink-agent | codex | claude-code | custom
└─ sessionId / messages / artifacts / status
```

运行链路原则：

- 会话消息流属于 App 运行时，不属于某一个可见 UI 面板。
- 右侧 Agent 面板、工作会话 Tab、未来的 Terminal Tab 都只是会话的不同视图。
- 隐藏右侧面板、切换为居中面板、关闭某个工作 Tab，都不应该导致后台流式消息丢失。
- UI 组件可以发起发送、中止、确认等动作，但事件订阅、消息落库、状态更新必须在全局运行层完成。

拷问与回答：

1. **右侧的新会话和工作空间里的新会话怎么区分？**  
   右侧按钮创建即时助手会话；工作空间 `会话` 分组里创建工作会话。文案不能都只叫“新会话”，后续应分别显示“新建助手会话”和“新建工作会话”。
2. **工作空间侧栏里的“会话”是不是右侧 Agent 的列表？**  
   不是。它是当前工作空间归属下的工作会话列表。右侧 Agent 可以临时协作，但只有提升/创建为工作会话后，才进入该列表。
3. **远程会话能不能叫 CCLink 会话？**  
   不能。产品上叫“会话”，展示元信息为运行位置和连接通道，例如 `远程 · CCLink · supermicro`。CCLink 只是 transport。
4. **IM 进来以后会不会又混乱？**  
   人聊人是消息；AI 处理任务才可能生成或升级为工作会话。IM 不是 conversation Tab 的同义词，避免把聊天列表和工作现场混在一起。
5. **“打开为工作会话”和“提升为工作会话”是不是一回事？**  
   第一版不是。现在先提供同一会话的工作区视图，保证可逆；等侧栏工作会话列表和删除/归档语义稳定后，再决定是否做真正的迁移。

## Activity Bar

Activity Bar 只放用户会反复切换的一级内容库或系统区。

建议保留：

| 入口     | 定位         | 说明                                                  |
| -------- | ------------ | ----------------------------------------------------- |
| 工作空间 | 核心内容组织 | 本地/远程工作空间、文件、会话、草稿、未归档           |
| 搜索     | 跨内容查找   | 当前工作空间优先，后续扩展到全局                      |
| 浏览器   | 网页资料库   | 书签、历史、下载、网页会话、常用网站                  |
| 消息     | IM 内容库    | 未来好友、群聊、工作分享；未实现前可隐藏              |
| 设置     | 系统管理     | 账号、模型、同步、远程连接、Agent 后端、Android、诊断 |

不建议作为一级入口：

| 功能                  | 去处                          | 原因                                           |
| --------------------- | ----------------------------- | ---------------------------------------------- |
| CCLink / 远程连接账号 | 设置 > 远程连接 / 账号        | 是连接通道与账号配置，不是日常内容库           |
| Android 管理          | 设置 > 设备 / Android         | Android 实例本质上是工作台 Tab，不是独立内容库 |
| 同步配置              | 设置 > 同步                   | 高频状态可进 Status Bar，配置不占一级侧栏      |
| 账号迁移和诊断        | 设置 > 账号 / 远程连接 / 诊断 | 属于异常处理和配置流程                         |

拷问标准：

> 如果用户每天不是为了“浏览这类内容”而切换它，就不要放 Activity Bar。

## Home

Home 是总入口和默认界面，不是一个普通工作空间。

Home 回答：

- 我可以从哪里继续？
- 有什么未处理？
- 我现在可以快速开始什么？

Home 大面内容：

- 最近工作空间。
- 最近 Tab / 会话 / 草稿。
- 未归档内容和待确认任务。
- 快速开始：新建草稿、打开网页、新建会话、打开工作空间。

约束：

- Home 不承载复杂管理。
- Home 不替代工作空间面板。
- Home 不展示大量设置项。

## 工作空间面板

工作空间面板是 DeepInk 最核心的侧栏。

结构：

```text
工作空间
├─ [本地] GAME
├─ [本地] DeepInk
├─ [远程 · 直连] Mac mini /Users/app/project-a
├─ [远程 · CCLink] supermicro /data/research
└─ 打开工作空间...

当前工作空间内容
├─ 文件      ← IDE 风格文件树，必须保留
├─ 会话      ← 当前工作空间下的会话
└─ 草稿      ← 当前工作空间下未保存内容

未归档
├─ 临时草稿
└─ 全局会话
```

产品规则：

- 本地和远程作为 badge，不新增“本地/远程”一级分组。
- 远程连接路线只体现在 badge 或副标题中，例如 `直连`、`CCLink`、`VPN`，不额外增加目录层。
- 点击工作空间就是切换当前归属。
- 同一时间重点展开当前工作空间，避免侧栏爆炸。
- 文件树是当前工作空间的内容浏览方式，不是被工作空间列表取代。
- Tab 列表不放侧栏，仍由 Workbench 顶部 Tab Bar 管理。
- 未归档固定在底部，承接没有明确归属的内容。

添加入口：

```text
添加工作空间
├─ 打开本地文件夹
├─ 添加远程工作空间
│  ├─ 直连服务器
│  └─ 通过 CCLink
└─ 新建临时草稿
```

添加成功后，用户回到工作空间列表；连接管理页不是日常停留页。

## 浏览器面板

浏览器面板可以保留，但它不是“打开浏览器 Tab”的唯一入口。

定位：

```text
浏览器面板 = 网页资料库
浏览器 Tab = 当前打开的网页
```

大面内容：

- 书签。
- 历史。
- 下载。
- 保存的网页会话。
- 常用网站。
- 未来的网页自动化任务记录。

约束：

- 浏览器面板不承担网页渲染，网页渲染在 Tab 中。
- 浏览器面板不塞 Agent 对话。
- 浏览器面板可以晚一点细化，不影响工作空间主线。

## Workbench Tabs

Workbench 是操作现场，所有具体工作都以 Tab 承载。

Tab 类型包括：

- Markdown / 文档编辑。
- 浏览器。
- Android。
- Terminal。
- 远程文件只读预览。
- 会话。
- 设置。
- 预览。

规则：

- 工作空间 Tab 跟随当前工作空间切换。
- 设置、Home、账号等全局 Tab 不应被工作空间切换误关。
- 会话 Tab 不以 CCLink、直连、Codex 命名；这些只作为运行元信息展示。
- Android 和浏览器一样，是工作台能力，不是一级导航对象。
- Terminal 和浏览器、Android 一样，是当前工作空间的工作现场；远程 Terminal 必须绑定远程工作空间和权限确认。
- 远程文件第一阶段只读，不能暗示已经支持远程编辑保存。

## 右侧 Agent 面板

右侧 Agent 面板是协作层，不是导航层。

它负责：

- 与当前工作空间/当前 Tab 协作。
- 展示执行状态、工具调用、确认请求。
- 承接 Agent 通知和必要的人机确认。
- 未来与 IM 消息流合并。

约束：

- Agent 面板不承载工作空间列表。
- Agent 面板不承载账号配置。
- Agent 面板不替代 Home。

## Settings

Settings 是能力和账号的归宿。

应收纳：

- 账号和订阅。
- 模型和 Agent 后端。
- 同步。
- 远程连接：直连 Remote、CCLink、设备配对、服务器同步、实时链路、诊断。
- Agent 后端：DeepInk Agent、Codex、Claude Code、自定义后端的选择与凭证。
- Android：SDK、模拟器、物理设备、安装状态。
- 浏览器：默认行为、权限、下载。
- 编辑器：Markdown、快捷键、外观。
- 诊断：日志、版本、云函数状态、开发期工具。

约束：

- Settings 可以有诊断按钮，但日常侧栏不应该像诊断台。
- 账号异常流程要可见、可恢复，但不挤占主工作区。

## Status Bar

Status Bar 只显示轻量、持续的状态。

适合显示：

- 当前工作空间来源：本地 / 远程 / 未归档。
- Agent 连接状态。
- 同步状态。
- 远程链路状态。
- 当前 Tab 类型。

不适合显示：

- 复杂错误详情。
- 多步骤配置流程。
- 账号迁移表单。

## 迁移策略

从现有界面迁移到上述形态时，先改信息架构，再补完整功能页。

第一阶段：收敛 Activity Bar。

- 保留 `工作空间 / 搜索 / 浏览器 / 设置`。
- `消息` 先不展示或保持隐藏，等 IM 主线开始后再开放。
- 移除 `远程 Agent / Android / 同步` 一级入口。
- Android 仍可通过 Tab Bar 的新建菜单或命令打开。
- 同步状态保留在 Status Bar；配置入口进 Settings。

第二阶段：把配置流迁进 Settings。

- 新增 `设置 > 远程连接`，承接直连 Remote、CCLink 身份、旧账号导入、服务器同步、实时链路和诊断。
- 新增或整理 `设置 > Agent 后端`，承接 Codex、DeepInk Agent、Claude Code 和自定义后端。
- 新增或整理 `设置 > 设备 / Android`，承接 SDK、模拟器、物理设备和安装状态。
- 保留 `设置 > 同步`，作为 WebDAV / 云同步配置页。
- 原侧栏里的调试按钮如“生成示例数据 / 清空”只能出现在开发诊断区，不能出现在日常主界面。

第三阶段：让工作空间面板承接远程工作。

- 远程服务器不作为一级导航对象。
- 远程服务器上报的目录变成 `[远程 · 通道] 设备名 / 路径` 工作空间。
- 点击远程工作空间后，展开该工作空间的文件树和会话。
- 没有工作空间归属的会话先落到未归档，避免丢失。
- Terminal 作为远程工作空间的 Tab 类型接入，不放入 Activity Bar。
- 不做单独“远程 Agent 面板”；远程会话、远程文件和远程 Terminal 都归入当前远程工作空间。

第四阶段：补 Home 和浏览器资料库。

- Home 负责继续工作、待处理和快速开始。
- 浏览器面板逐步补书签、历史、下载和网页会话，不阻塞工作空间主线。

迁移约束：

- 不一次性重写 Settings。
- 不在迁移期删除底层 CCLink / Android 能力，只移动入口。
- 不把远程文件编辑伪装成已完成；第一阶段只读。
- 不让用户失去打开 Android Tab、同步或添加远程工作空间的路径。

## 暂不细化的部分

以下内容先不在本文档钉死：

- 浏览器面板里的具体书签/历史交互。
- 消息/IM 的完整页面形态。
- 远程文件编辑和保存策略。
- 远程 Terminal 的权限、安全边界和 shell 生命周期。
- Android 多实例管理。
- 设置页每个分组的完整字段。
- Home 的最终视觉排版。

## Grilling

1. 如果继续把 CCLink、Android、同步都放进 Activity Bar，DeepInk 会像调试工具集合，而不是工作入口。
2. 如果工作空间面板不保留文件树，会丢掉 IDE 用户最熟悉的浏览方式。
3. 如果浏览器面板直接删掉，会损失网页资料库的长期价值；正确做法是保留但降级为资料管理。
4. 如果远程服务器作为一级对象，用户会被迫理解基础设施；正确主对象应是远程工作空间。
5. 如果 Settings 不收纳账号和设备问题，主界面会长期被异常流程污染。
6. 现在只应冻结页面职责，不应冻结所有细节；否则很快会因 IM、远程编辑、多设备而返工。
7. 如果把 Remote 继续等同 CCLink，用户会误以为没有 CCLink 就不能用远程；这会直接削弱直连服务器、内网、VPN 和专业用户场景。
8. 如果给远程再加“服务器”父层，侧栏会回到基础设施导航；本地和远程只能是 badge，不应成为目录层。
9. 如果 Codex 被设计成工作空间类型，会把“在哪里工作”和“谁来执行”混在一起；Codex 应该是执行后端。
10. 如果 Terminal 只做按钮不做工作空间归属和权限边界，就会变成半吊子 IDE；必须先定义它属于哪个工作空间、在哪台机器执行、如何关闭和恢复。
11. 如果保留“远程 Agent 面板”，会把远程会话特殊化；真正特殊的是运行位置和连接通道，不是会话形态。
